# HELIos Idempotency Standard

## Scope

Applies to any webhook, queue consumer, scheduler, or API endpoint that can be retried or redelivered.

## Required Pattern

1. Compute deterministic idempotency key from business identity, not transport metadata.
2. Attempt atomic claim using a dedupe store before side effects.
3. Persist status transitions (`processing`, `completed`, `failed-permanent`) with timestamps.
4. Return previously computed result for duplicate keys when available.
5. Expire dedupe keys using TTL policy aligned with replay window.

## Key Construction Guidance

- Prefer key shapes such as:
  - `provider:eventType:resourceId:eventVersion`
  - `workflow:action:entityId:logicalTimestamp`
- Avoid random UUIDs generated per attempt.
- Include enough entropy to avoid collisions between semantically different actions.

## Storage Options

| Backend | Claim Primitive | Notes |
|--------|------------------|------|
| Redis | `SET key value NX EX ttl` | Fast path, ensure persistence and replication policy |
| Postgres | Unique index on `idempotency_key` | Use status columns and cleanup job for retention |
| DynamoDB | Conditional write + TTL | Good for high throughput distributed systems |

## Anti-Loop Controls

- Tag automation-originated writes (`origin=helios-automation`).
- Ignore or down-prioritize events generated by your own writeback path when safe.
- Maintain per-resource loop counter with cooldown windows.
- Add circuit breaker when repeated reciprocal event patterns are detected.

## Acceptance Tests

1. Same event payload delivered N times produces one side effect.
2. Out-of-order duplicates still return consistent final state.
3. Concurrent duplicate deliveries are serialized by claim primitive.
4. Duplicate suppression does not block distinct events with close timestamps.
